## 1. LWCの簡単な特徴
HTML、JavaScript、CSSのモダンなWEB標準機能をベースにしている。

### 1.1. コンポーネントベース
小さい再利用なコンポーネントを作成して、組み合わせながらUIを構築する
※Reactを知っている人であれば、コンセプトは大体同じという理解でOK

### 1.2. 開発環境の制限あり
一般的？なJavaScriptのフレームワーク（React/Vue/Anglar）は
色々な開発ツールを利用して比較的自由に開発環境を構築可能

しかしLWCはSalesforce CLIなどを利用する必要があるため
開発環境はある制限がある（大体、Salesforce内の開発ツール or VSCode）

### 1.3. ライブラリ、フレームワークに近い
LWCは正確にはライブラリでもフレームワークでもなく、プログラミングモデル…
みたいな話があるけれど、初学者からみるとReactとかの位置づけに近いもの

HTML、JavaScript、CSSの標準をベースにしているけれど
Salesforceの多くの機能と密接に統合できるもの
⇒公式のサンプル見ながらコンポーネントを作っていくと
　SalesforceのUIに馴染んだものが、WEB標準を知っていれば自然と作れる…はず

[プログラミングモデルの下りの参考](https://qiita.com/hrk623/items/70fe3bc161a30b54686d)

### 1.4. JavaScriptのテストにJestが利用可能
JestはFacebookが開発したJavaScriptの「テストフレームワーク」
FacebookはJavaScriptフレームワークとしてReactも提供しているので（正確にはフレームワークじゃないけど
ReactのプロジェクトではJestが利用されることがそこそこあるっぽい

JavaScriptのテストフレームワークはJest以外にも
Mocha、Jasmine、QUnitなど色々存在している

SalesforceでもJest以外のテストフレームワークは利用できるっぽいけれど
SalesforceはJestを正式にサポートしており特化した環境が整っているので


### 1.5. Apexとの比較
- ローカル開発が可能
ApexはSalesforceサーバ上で動作するため、一度デプロイする必要がありましたが
LWCはWeb標準機能がベースになっているのでローカル開発が可能
※Webページ（LWC）からApexを呼び出すパターンの場合は
　Apex側はデプロイしておく必要あり
※JavaScript側はテストコードを書けばローカルでデバッグできるが
　Webページ側はローカルサーバ起動というひと手間が必要になる模様

**追記**
LWC用のローカルサーバは、どうやらベータ版だった模様
現状、過去の手順通りに導入しようと思っても中々エラーが解決しない。
たぶん解決方法はある気がするけれど、時間がかかりそうなのでいったんスルー

- リリース条件は特にない…はず
Apexはカバレッジが75%以上のテストを実施しないと本番環境へリリース出来ない制約があったが
LWCに関してはJavaScript側のテストコードを書かなくてもリリース出来る…はず
